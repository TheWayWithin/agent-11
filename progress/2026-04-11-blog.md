# Why was I calling OpenAI from inside Claude Code?

I stared at the Python script for about four seconds before the question formed.

`enhance_dailyreport.py`. Six hundred lines. Lived in agent-11's `/dailyreport` command. Took the day's progress log, shipped it to gpt-4o-mini via the OpenAI API, got back a blog post and two social posts, wrote them to disk. Cost about $0.002 per report. Had been there for months.

I was running it from inside Claude Code. Opus 4.6 was literally sitting in the chair.

How had I not noticed this?

The answer is dull: historical accident. When the script was first written, slash commands were simpler. You shelled out to Python, Python did the work, gpt-4o-mini was the obvious choice: cheap and already wired in. You stop thinking about the architecture the moment it runs.

But we'd just spent three hours bolting a 1500-word voice guide onto the script — forty-odd banned AI-tell words, rules against rule-of-three adjective stacks, British spelling, a pub test. gpt-4o-mini kept drifting. It would catch half the banned vocabulary and miss the other half. Opus, when I tested it in a separate session, held the rules cold.

The better writer was in the room. I was paying another model to do worse work.

So we deleted it. Six hundred lines of Python, gone. `OPENAI_API_KEY` disappeared from `.env.mcp.template`. A new ten-step Claude-native protocol replaced the old script: read `progress.md`, load the voice guide, draft the long-form post, derive the Twitter and LinkedIn versions, run a voice scrub pass, write the files.

That last bit matters more than it sounds. The old Python script could only warn about AI-tell words after generation, because it had no way to call back to gpt-4o-mini for a fix. The Claude-native version has full edit control. If a banned word sneaks into the draft, the scrub rewrites the sentence on the spot. No warning, no "please review". Just a fix.

---

While I was in there, a second thing fell out. BOS-AI uses the same `/dailyreport` command, so I tried installing the updated version from my local agent-11 working copy before pushing to main. The install script detected "remote" mode, tried to pull files from GitHub that weren't yet there, and rolled back cleanly.

Why was local mode not triggering? Line 11 of `install.sh`:

```bash
PROJECT_ROOT="$(cd "$SCRIPT_DIR/../.." && pwd)"
```

The script lives three directories below the repo root. `../..` goes up two. The `.git` check was pointing at `agent-11/project/`, not `agent-11/`. It had been wrong since whenever this script was first written. Nobody caught it because the common install path is curl-piped from GitHub, which doesn't go through the local-mode branch at all. The bug had been latent in a codepath nobody exercised.

One character. `../..` became `../../..`. Every local install works again.
